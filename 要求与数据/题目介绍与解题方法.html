PCB 布局优化大作业题目介绍与解题方法

1. 文档目的

这份文档基于当前工作区中的 4 类材料整理而成：

  • 布局优化大作业/大作业说明.md
  • 布局优化大作业/data/ 下的 small-1 到 small-5 数据集
  • 布局优化大作业/visualize_pcb.py
  • 选题二 面向多目标约束的PCB布局启发式优化算法研究.pptx

目标是把题目要求、输入输出格式、推荐实现步骤、可行算法路线和实验报告写法整理成一份可直接照着做的说明。

2. 题目到底要做什么

从题目说明看，这次大作业的核心任务是：

在给定 PCB 元器件、网络连接和初始布局的前提下，调整元器件位置，使总线长指标 HPWL 尽可能小，同时满足布局合法性约束。

可以把它理解成一个带约束的组合优化问题：

  • 决策变量：每个元器件的位置；朝向字段按输入 *.pl 保留，不作为本次作业的主要优化变量
  • 优化目标：最小化所有网络的 HPWL
  • 约束条件：
    • 元器件之间至少满足最小间距
    • 所有元器件必须放在允许的画布范围内
    • 输出布局文件格式必须与输入 *.pl 一致

需要注意两点：

  1. pptx 的题目背景是“面向多目标约束的 PCB 布局启发式优化”，范围更大，包含 DRC、总线长度、SIPI 等更广义目标。
  2. 当前这次大作业的实际落地版本已经简化为“以 HPWL 为主目标，带最小间距和边界约束”的布局优化问题。

所以实现时不要把问题做得过宽。先把单目标 HPWL 优化做扎实，比空泛谈多目标更重要。

3. 数据文件怎么理解

每个数据集由 3 个文件组成：

  • *.nodes：元器件尺寸
  • *.nets：网络连接和引脚相对偏移
  • *.pl：元器件初始位置与朝向

3.1 .nodes

这一类文件定义元器件名、宽度和高度，例如：

  1. R96_3 5 10
  2. R52_3 10 5
  3. U2_3 71 71

解释如下：

  • R96_3、R52_3 是普通阻容器件，尺寸较小
  • U2_3 是大尺寸芯片，通常是布局中的关键器件

程序中建议解析成：

  1. name -> {width, height}

3.2 .nets

这一类文件定义每个网络包含哪些引脚，以及引脚相对于元器件中心的偏移量，例如：

  1. NetDegree : 7 NET_3V3_CTRL_HVDD_DVDD_PL0_3
  2.     R37_3 I : 4 0
  3.     R38_3 I : 4 0
  4.     R40_3 I : -4 0
  5.     R56_3 I : 0 4
  6.     C19_3 I : -4 0
  7.     C20_3 I : -7 0
  8.     U2_3 I : -34 -18

其中：

  • NetDegree 表示这个网络有多少个引脚
  • (dx, dy) 表示该引脚相对元器件中心的偏移

程序中建议解析成：

  1. net -> [ (component, dx, dy), ... ]

3.3 .pl

这一类文件给出元器件左下角坐标和朝向，例如：

  1. R96_3 2500 1773 : W
  2. R52_3 3762 2031 : N
  3. U2_3 3693 2056 : E

这里有两个关键点：

  • (x, y) 是元器件矩形的左下角，不是中心点
  • N/E/S/W 表示朝向；本次作业的 HPWL 计算按题面给出的公式处理，即使用元件左下角、宽高和引脚相对中心偏移计算引脚绝对坐标

程序中建议解析成：

  1. name -> {x, y, orient}

4. 目标函数怎么计算

题目要求使用 HPWL 作为优化目标：

HPWL = \sum_{net} [(\max x_i - \min x_i) + (\max y_i - \min y_i)]

其中 (x_i, y_i) 是某个网络中第 i 个引脚的绝对坐标。

引脚绝对坐标的基本计算方式是：

x_{pin} = x_{comp} + \frac{w}{2} + dx

y_{pin} = y_{comp} + \frac{h}{2} + dy

本次作业说明和当前工作区里的 visualize_pcb.py 都按上面的公式直接使用原始 dx, dy，没有要求根据 N/E/S/W 对引脚偏移做旋转变换。因此求解器应优先严格对齐题面公式，保证 HPWL 计算口径和作业要求一致。

如果后续想把方案扩展成更完整的 PCB 布局模型，可以把朝向旋转、宽高互换和引脚偏移变换作为扩展项，但不要把它作为本次作业的必要实现。

5. 约束条件怎么检查

5.1 最小间距约束

题目要求任意两个元器件至少保留 2 个单位的间距。题目给出的判定方式本质上是：

  • 计算两个矩形在 x 方向上的间距 gap_x
  • 计算两个矩形在 y 方向上的间距 gap_y
  • 当且仅当 gap_x >= 2 或 gap_y >= 2 时，这两个器件的相对位置合法

直观理解就是：

  • 两个矩形可以在某一个方向上靠得比较近
  • 但必须至少在一个方向上拉开足够的安全距离

程序里最稳妥的做法不是直接背公式，而是统一写成矩形合法性判断函数：

  1. 先得到器件 A、B 的左右上下边界
  2. 计算两个方向上的分离量
  3. 判断是否满足最小间距

建议单独封装：

  1. is_non_overlapping_with_gap(comp_a, comp_b, min_gap=2)

5.2 边界约束

题目要求所有元器件都放在给定画布范围内。

当前数据文件里没有看到单独的 board outline 文件，因此实现上通常有两种做法：

  • 以原始布局的外接矩形作为合法画布范围
  • 在代码里固定一个已知边界框

更稳妥的写法是：

  • 默认从原始 *.pl 推导布局边界
  • 在代码中预留参数，允许手动指定边界

这样既能兼容当前作业，也便于后续实验调参。

需要在报告中说明：如果采用原始布局外接矩形作为画布边界，这是基于当前数据文件缺少独立 board outline 的实现假设。

6. 这道题推荐怎么做

从作业规模看，最实用的路线不是复杂的精确算法，而是“启发式初始化 + 局部搜索/模拟退火 + 合法性修复”的组合方法。

推荐主线如下：

  1. 先完成数据解析与 HPWL 计算
  2. 再完成合法性检查
  3. 然后实现一个能输出新 *.pl 的基线优化器
  4. 最后再用更强的搜索策略去压低 HPWL

一个比较稳、也比较好写报告的方法是：

  • 基线：贪心压缩或简单局部搜索
  • 主方法：模拟退火
  • 可选增强：按网络关系做聚类移动，或引入交换操作

7. 推荐实现步骤

下面这套步骤基本可以直接作为你的开发顺序。

第一步：写解析器

先把 3 类文件都读出来，建立统一的数据结构：

  • components[name] = {width, height}
  • placements[name] = {x, y, orient}
  • nets = [{name, pins}]

这一步完成后，先打印：

  • 元器件数
  • 网络数
  • 总引脚数
  • 最大器件尺寸

如果这些统计值和题面一致，说明解析基本没问题。

当前工作区中已经能确认的规模如下：

数据集 元器件数 Nets Pins
small-1 57 85 167
small-2 43 82 129
small-3 41 49 105
small-4 46 26 85
small-5 115 199 334

其中 small-5 是示例数据，不计分。

第二步：写 HPWL 计算器

建议先实现两个层次的接口：

  1. calc_net_hpwl(net, placement)
  2. calc_total_hpwl(all_nets, placement)

这样做的好处是：

  • 便于调试单条网络
  • 后面做增量更新时也方便复用

这一步完成后，可以先验证 small-5 的初始布局。按当前数据计算，small-5_original_artifacts 的总 HPWL 约为 9075.0，参考优化结果 small-5_optimized 约为 2882.0。这说明题目中的优化空间非常大。

第三步：写合法性检查器

至少要有下面几个函数：

  1. check_boundary(component, placement, board_bbox)
  2. check_pairwise_gap(component_a, component_b, placement, min_gap)
  3. check_layout_legality(all_components, placement, board_bbox, min_gap)

建议一开始先用朴素的两两检查，因为数据规模不大，足够跑通。

如果后面搜索迭代很多，再考虑：

  • 网格索引
  • 扫描线
  • 邻域候选筛选

第四步：先做一个可运行的基线方法

不要一上来就写复杂退火。先做一个简单但可运行的方法：

  1. 从原始布局出发
  2. 随机选择一个元器件
  3. 尝试向四周小范围移动
  4. 如果合法且 HPWL 下降，就接受
  5. 重复若干轮

这个版本的意义不是拿最好结果，而是尽快打通：

  • 输入读取
  • 目标函数
  • 约束检查
  • 输出 *.pl

只要这条链路通了，后续换算法就很容易。

第五步：加入更有效的邻域操作

推荐至少准备 3 类动作：

  1. 平移：把某个器件移动到附近位置
  2. 交换：交换两个器件的位置
  3. 聚类拉近：把与高连接度器件相连的器件向其中心靠拢

为什么这三类动作有效：

  • 平移适合微调
  • 交换适合跳出局部结构
  • 聚类拉近有利于快速降低长网络的跨度

如果时间有限，优先顺序是：

  1. 平移
  2. 交换
  3. 聚类拉近

第六步：主算法用模拟退火

模拟退火很适合这类离散布局问题，因为：

  • 允许在早期接受少量变差解，避免卡死在局部最优
  • 实现难度适中
  • 报告也容易讲清楚

推荐的退火框架如下：

  1. 初始解设为原始布局或基线优化后的布局
  2. 设定初始温度 T0
  3. 每轮随机生成一个邻域动作
  4. 若新解更优则接受
  5. 若更差，则按概率 exp(-delta / T) 接受
  6. 温度按比例衰减
  7. 直到达到终止条件

常见终止条件：

  • 温度低于阈值
  • 连续若干轮无改进
  • 达到最大迭代次数

第七步：做增量代价计算

这是后期最值得投入的优化点。

如果每次移动一个器件都重新计算全局 HPWL，会浪费很多时间。更合理的做法是：

  1. 预先记录每个器件属于哪些网络
  2. 当某个器件移动时，只重算受影响的那些网络
  3. 用“新网络 HPWL 之和 - 旧网络 HPWL 之和”得到增量代价

这样通常能把迭代效率提高一个数量级。

第八步：加入合法性修复

搜索过程中经常会出现：

  • 出界
  • 与别的器件间距不足

推荐两种处理方式：

  • 严格模式：非法解直接拒绝
  • 修复模式：先投影回边界，再尝试局部挪开冲突器件

作业实现里建议优先使用严格模式，因为：

  • 逻辑简单
  • 调试成本低
  • 结果更容易解释

如果后面想进一步提升效果，再引入修复机制。

第九步：输出并验证结果

优化完成后，需要输出新的 *.pl 文件，并完成 3 类验证：

  1. HPWL 是否降低
  2. 布局是否合法
  3. 可视化结果是否合理

当前工作区自带 visualize_pcb.py，可直接用于生成布局图，对写报告很有帮助。

8. 一条比较稳的算法路线

如果你希望方案既容易实现，又能写出像样的报告，建议采用下面这条路线：

方案名称

基于模拟退火的 PCB 元器件布局优化方法

核心思想

  1. 用 Bookshelf 数据建立元器件、引脚和网络模型
  2. 用 HPWL 衡量布局优劣
  3. 用最小间距和边界条件保证可行性
  4. 用平移和交换作为局部搜索动作
  5. 用模拟退火控制搜索过程，平衡探索与收敛

这条路线的优点

  • 代码量可控
  • 思路清晰
  • 容易做参数实验
  • 容易画出收敛曲线
  • 容易解释为什么有效

这条路线的缺点

  • 参数较多，需要调温度和迭代次数
  • 如果邻域设计太弱，收敛速度会偏慢
  • 如果合法性检查太慢，整体运行时间会上升

这几个缺点都可以在报告里作为“后续改进方向”来写。

9. 写代码时最容易踩的坑

9.1 HPWL 计算口径不一致

本次作业的 HPWL 应按题面公式计算：

  1. x_pin = x_comp + w_comp / 2 + dx
  2. y_pin = y_comp + h_comp / 2 + dy

不要在主实验里自行加入未说明的朝向旋转规则，否则可能导致你计算出的 HPWL 和作业说明、可视化脚本、报告对比口径不一致。

9.2 误把 (x, y) 当成中心坐标

题目里的 (x, y) 是左下角，不是中心点。这个错误会直接导致：

  • 引脚位置算错
  • HPWL 算错
  • 边界检查算错

9.3 合法性判断写反

最小间距约束不是简单的“不重叠”，而是“至少在一个方向上分开 2 个单位”。这比普通矩形不重叠更严格。

9.4 每次都全量重算

小数据集还能跑，但一旦迭代多了，效率会迅速变差。建议至少在第二版实现增量重算。

9.5 只看数值不看图

HPWL 下降不代表布局一定正常。可视化能快速暴露：

  • 器件堆叠
  • 异常散布
  • 某个器件被挤到边界角落

所以最终一定要看图。

10. 报告可以怎么写

实验报告建议按照下面的结构组织。

10.1 问题定义

写清楚：

  • 输入是什么
  • 输出是什么
  • 目标函数是什么
  • 约束是什么

10.2 方法描述

重点说明：

  • 数据结构
  • HPWL 计算方法
  • 邻域动作设计
  • 模拟退火流程
  • 合法性检查方法

10.3 实验设置

给出：

  • 初始温度
  • 降温系数
  • 每温度迭代次数
  • 最大迭代轮数
  • 最小间距参数
  • 运行环境

10.4 结果展示

对每个数据集列出：

  • 初始 HPWL
  • 优化后 HPWL
  • 降低比例
  • 运行时间

建议用表格表示。

10.5 可视化与分析

插入优化前后布局图，分析：

  • 高连接器件是否更集中
  • 长距离网络是否减少
  • 哪类元器件移动最明显
  • 算法是否出现局部拥挤现象

10.6 不足与改进

可以写：

  • 目前只优化 HPWL，尚未显式纳入更多电性能指标
  • 合法性检查仍是朴素实现
  • 邻域动作还可以更丰富
  • 后续可扩展为多目标优化或自适应参数调优

这样就能自然呼应 pptx 中更大的研究方向。

11. 建议的最终提交内容

建议至少准备以下文件：

  • 主程序，例如 main.py 或 optimize_pcb.py
  • 数据解析模块
  • HPWL 与合法性检查模块
  • 优化后的 *.pl 文件
  • 可视化结果图
  • 实验报告 PDF
  • 简短运行说明

运行说明建议写清：

  • Python 版本
  • 依赖库
  • 执行命令
  • 输出文件位置

11.1 关于加分项和高分项

当前 大作业说明.md 没有写明单独的“加分项”。但从报告要求看，下面这些内容会明显提升完成度：

  • 对 4 个计分数据集都给出优化前后 HPWL 对比
  • 记录运行时间、迭代次数和主要参数
  • 给出优化过程的收敛曲线
  • 做不同方法或不同参数的对比实验
  • 插入优化前后的布局可视化图，并解释 HPWL 下降是否对应更合理的布局

也就是说，真正应该优先补强的是“结果对比、参数分析、收敛过程和可视化分析”，而不是脱离题面额外扩展复杂约束。

12. 一句话总结

这道题最稳的做法不是追求花哨，而是按“解析数据 -> 正确计算 HPWL -> 严格检查合法性 -> 用模拟退火做局部搜索 -> 输出结果并可视化验证”这条链路一步一步做扎实。只要这五件事做完整，代码、结果和报告都会比较顺。
